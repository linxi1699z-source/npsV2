# 后台 PRD 写作规范

来源：`/Users/xiemin/Downloads/支付管理后台PRD(续订）.md`。本文用于后续编写后台管理系统、支付、会员、任务配置、问卷、订单、权益、账单等业务 PRD。

## 1. 文档整体结构

PRD 必须使用以下一级标题，顺序固定：

1. `# 版本信息`
2. `# 文档说明`
3. `# 需求背景`
4. `# 目标与关键指标（KPI）`
5. `# 功能详细说明`
6. `# 非功能需求`
7. `# 多语言`
8. `# 埋点`
9. `# 产品项目规划（从调研、计划到转入开发）`
10. `# 审批与签名`

文档开头可先放文档标题，如 `# NPS管理后台PRD`。标题后进入 `# 版本信息`。

输出交付要求：
- 默认输出为 `.docx` 文件，除非用户明确要求 Markdown、PDF 或其他格式。
- 生成 `.docx` 时仍需保持本文规定的标题层级、表格结构、字段说明、交互说明、状态流转、重复拦截和幂等规则。
- 生成 `.docx` 前可先用 Markdown 或结构化草稿组织内容，但最终交付物必须是可直接打开和评审的 Word 文档。

## 2. 版本信息

保留版本号、创建日期和变更日志。

```markdown
# 版本信息

版本号：V1.0

创建日期：YYYY-MM-DD

## 变更日志

| 时间 | 版本号 | 变更人 | 主要变更内容 |
|---|---|---|---|
| YYYY-MM-DD | V1.0 | 姓名 | 初建 |
```

变更日志要求：
- 时间使用 `YYYY-MM-DD`。
- 主要变更内容可用 `<br>` 分行。
- 版本号按 `V1.0`、`V1.1`、`V1.1.1` 递增。

## 3. 文档说明

该章节主要做名词解释，不写功能细节。

```markdown
# 文档说明

## 名词解释

| 术语 / 缩略词 | 说明 |
|---|---|
| 术语 | 说明 |
```

名词解释要求：
- 解释业务术语、英文缩写、外部系统名、状态名、关键 ID。
- 对会影响字段、状态或接口判断的术语，说明业务边界。

## 4. 需求背景

固定二级标题：

```markdown
# 需求背景

## 产品 / 数据现状调研

## 用户调研

## 竞品分析
```

写法要求：
- `产品 / 数据现状调研` 写当前业务问题、现有流程、数据现状。
- `用户调研` 写使用者、痛点、典型场景；无调研时写 `待补充`。
- `竞品分析` 可放链接或结论；无竞品时写 `待补充`。

## 5. 目标与关键指标（KPI）

使用目标和 KPI 两组项目符号：

```markdown
# 目标与关键指标（KPI）

- 目标：
  1. ...

- KPI：
  1. ...
```

要求：
- 目标描述业务达成结果。
- KPI 尽量可衡量，如配置效率、错误率、人工处理时长、投放成功率、数据一致性。

## 6. 功能详细说明

必须包含四个二级标题：

```markdown
# 功能详细说明

## 5.1 主要功能范围

## 5.2 产品流程图

## 5.3 交互原型

## 5.4 功能说明
```

### 6.1 主要功能范围

用表格列举模块、功能点、备注。

```markdown
| 模块 | 功能点 | 备注 |
|---|---|---|
| 模块名 | 功能点 | 说明 |
```

要求：
- 模块粒度按导航或业务域划分。
- 功能点覆盖列表、新增、编辑、查看、启用/禁用、导入/导出、状态更新、任务处理、回调/同步等。

### 6.2 产品流程图

写流程图链接或 Mermaid 流程图。缺失时写 `待补充流程图链接`。

流程说明必须覆盖：
- 正常主流程。
- 异常流程。
- 状态流转关键节点。
- 需要预警或人工处理的节点。

### 6.3 交互原型

放原型链接即可，必要时按页面列出。

```markdown
## 5.3 交互原型

- 原型总入口：链接
- 页面 A：链接
- 页面 B：链接
```

### 6.4 功能说明

按模块和页面展开：

```markdown
### 5.4.1 模块名称

#### 页面名称

页面目标：...

入口：...

权限：...
```

每个页面至少说明：
- 页面目标和使用角色。
- 页面入口和跳转来源。
- 搜索条件。
- 顶部按钮。
- 列表字段。
- 表单字段。
- 弹窗交互。
- 状态流转。
- 页面跳转逻辑。
- 字段限制。
- 重复拦截。
- 幂等规则。
- 异常提示。

## 7. 字段说明表

推荐字段表结构：

```markdown
| 字段 | 中文 | 是否必填 | 类型 | 默认值 | 规则 / 交互说明 |
|---|---|---|---|---|---|
| task_id | 任务 ID | 是 | string/int | 系统生成 | 自增或唯一；列表倒序展示 |
```

字段写法要求：
- `字段` 使用后端或数据字段名，如 `task_id`、`status`、`created_at`。
- `中文` 使用页面展示名。
- `是否必填` 使用 `是 / 否 / 条件必填`。
- `类型` 使用 `string / int / boolean / enum / datetime / file / array / object`。
- `默认值` 明确写系统默认值或 `无`。
- `规则 / 交互说明` 写展示格式、校验、枚举、联动、权限、错误提示。

复杂字段可在单元格内用编号：

```markdown
1. 状态值：初始状态、投放中、暂停
2. 默认值：初始状态
3. 状态更新后需要同步列表和详情页
```

## 8. 页面交互说明

交互说明推荐表：

```markdown
| 触发位置 | 触发动作 | 前置条件 | 系统处理 | 结果 / 提示 |
|---|---|---|---|---|
| 新增按钮 | 点击 | 有新增权限 | 跳转新增页面 | 展示新增表单 |
```

必须覆盖：
- 按钮点击。
- 下拉选择。
- 搜索/重置/筛选。
- 新增/编辑/查看。
- 启用/禁用/暂停。
- 上传/删除文件。
- 弹窗确认/取消。
- 页面跳转。

弹窗说明推荐表：

```markdown
| 弹窗 | 触发条件 | 文案 | 按钮 | 确定逻辑 | 取消/关闭逻辑 |
|---|---|---|---|---|---|
| 暂停确认弹窗 | 点击暂停 | 确认暂停任务吗？ | 确定 / 取消 | 更新状态 | 关闭弹窗 |
```

## 9. 状态流转

状态流转必须独立成表，不要只写在字段说明中。

```markdown
| 当前状态 | 触发动作 | 前置条件 | 目标状态 | 说明 |
|---|---|---|---|---|
| 初始状态 | 提交投放 | 校验通过 | 投放中 | 开始投放任务 |
```

要求：
- 明确哪些状态允许操作。
- 明确不可操作时按钮隐藏、禁用或提示。
- 明确状态更新后的关联数据变化。

## 10. 重复拦截和幂等规则

重复拦截写法：

```markdown
#### 重复拦截

| 场景 | 判重字段 | 拦截规则 | 提示文案 |
|---|---|---|---|
| 新增任务 | task_name + scene | 有效数据中不允许重复 | 任务名称已存在 |
```

幂等规则写法：

```markdown
#### 幂等规则

| 场景 | 幂等键 | 处理规则 |
|---|---|---|
| 回调处理 | notification_id | 已处理则直接返回成功，不重复创建记录 |
```

必须考虑：
- 表单重复提交。
- 回调重复通知。
- 文件重复上传。
- 接口重试。
- 状态已变更后再次操作。
- 同一业务主键重复创建。

## 11. 接口 / 服务端逻辑说明

如 PRD 涉及服务端接口，用模块表或接口表描述。

```markdown
| 模块 | 说明 |
|---|---|
| 服务端 | 1. 接收入参...<br>2. 校验...<br>3. 返回... |
| 前端 / 后台 | 1. 调用接口...<br>2. 根据结果展示... |
```

接口说明至少包含：
- 入参。
- 出参。
- 校验规则。
- 失败重试。
- 异常提示。
- 数据落库或状态更新。

## 12. 非功能需求

固定二级标题：

```markdown
# 非功能需求

## 财务需求

## 运营或产品

## UI

## 研发
```

按业务补充安全、权限、性能、稳定性、审计、预警、兼容性、数据保留等要求。没有要求时写 `无` 或 `待补充`。

## 13. 多语言

写多语言字段、Key、语言范围、回退规则。

```markdown
# 多语言

1. 页面文案是否需要多语言：是 / 否
2. 多语言字段：
   - field_name
3. 回退规则：未配置时使用默认语言
```

如不涉及多语言，写 `本期不涉及多语言。`

## 14. 埋点

推荐表：

```markdown
| 埋点事件 | 触发时机 | 属性 | 说明 |
|---|---|---|---|
| event_name | 点击提交 | task_id, status | 统计提交结果 |
```

如不涉及埋点，写 `本期不涉及新增埋点。`

## 15. 产品项目规划

放项目管理链接、排期、里程碑或甘特图说明。

```markdown
# 产品项目规划（从调研、计划到转入开发）

> 项目管理文档附在此处，以下甘特图示一个例子，产品经理自行决定是否放在本文档
```

## 16. 审批与签名

固定表格：

```markdown
# 审批与签名

| 阶段 | 签名 / 审核人 | 评论要点 |
|---|---|---|
| PM评审（初评） |  |  |
| 初评PRD |  |  |
| PM评审（细评） |  |  |
| 细评PRD |  |  |
```

## 17. 写作风格

- 使用明确、可测试的规则，不写笼统表达。
- 后台页面说明优先按页面和操作拆分。
- 字段、状态、接口、回调、任务处理等数据逻辑必须写清楚。
- 对外部系统依赖写明系统名、接口来源、失败处理和重试策略。
- 涉及状态和金额等高风险数据时，明确审计、幂等、重试和异常预警。
- 未确认内容不要编造，统一写 `待补充`。

## 18. 输出验收清单

- 一级标题完整且顺序正确。
- `文档说明` 包含名词解释。
- `功能详细说明` 包含功能范围、流程图、交互原型、功能说明。
- 每个页面有字段表和交互表。
- 每个表单字段标注必填、类型、默认值和交互/校验说明。
- 弹窗说明包含触发、文案、按钮、确认和取消逻辑。
- 状态流转单独成表。
- 重复拦截和幂等规则单独说明。
- 页面跳转和权限边界明确。
- 多语言、埋点、非功能需求即使不涉及也要写明“不涉及”。
- 审批与签名表保留。
- 默认生成 `.docx` 交付文件，且文档结构与本规范保持一致。
